Project architecture is the decision you make before writing any code: how will the code be organized, which modules have which responsibilities, and how will data flow through the system. Getting this wrong early is expensive; getting it right makes every subsequent AI-assisted session faster and more reliable because the model always knows where code belongs.

> Project architecture = deciding "where code lives, how modules connect, and who owns what" before any business logic is written.

## Core design principles

Every template on this page follows the same five principles. These are not optional вЂ” they are what make AI-assisted development sustainable at scale.

<CardGroup cols={2}>
  <Card title="Separation of concerns" icon="layers">
    Each directory and module has one job. API code does not contain business logic. Business logic does not contain database calls. The layer structure makes the separation explicit.

    ```text
    API в†’ Service в†’ Repository в†’ Database
    ```
  </Card>
  <Card title="Interface first, implementation later" icon="plug">
    Define the interface (inputs, outputs, error cases) before writing any implementation. AI-generated code that targets a clear interface is dramatically more reliable than code written into ambiguous context.
  </Card>
  <Card title="Data and functions first" icon="database">
    Identify what data structures the system needs and what transformations it applies to them before deciding on modules or classes. Data flow defines the architecture; architecture does not define the data flow.
  </Card>
  <Card title="Configurability" icon="settings">
    Configuration lives outside code. The priority order is: environment variable в†’ config file в†’ default value. Nothing environment-specific is hardcoded.
  </Card>
  <Card title="Testability" icon="flask">
    Every module can be tested independently. External dependencies can be replaced. Functions have clear inputs and outputs with no hidden state.
  </Card>
  <Card title="Git-friendliness" icon="git-branch">
    Data, logs, and model files are excluded from version control. Only source code and configuration examples are committed.
  </Card>
</CardGroup>

## Module boundary principles

The most important architectural decision is how to split modules. The rule is: **split by responsibility, not by feature**.

### The layered architecture pattern

```text
Presentation layer  (UI / API endpoints)
       в†“
Service layer       (business logic, pure functions)
       в†“
Data access layer   (database queries, repository pattern)
       в†“
Database / external systems
```

**Rule**: upper layers call lower layers. Lower layers never depend on upper layers. This one rule prevents the majority of coupling problems.

### Module split example: trading system

```text
market_data/    вЂ” price feeds and data ingestion
strategy/       вЂ” signal generation logic
risk/           вЂ” position sizing and risk constraints
order/          вЂ” order submission and tracking
account/        вЂ” balance and P&L tracking
```

Each module:
- Does exactly one type of thing
- Exposes a defined interface
- Has no knowledge of the modules that call it

## New project checklist

Before writing any business logic, verify each item:

- [ ] `README.md` created with project description and setup instructions
- [ ] `LICENSE` file set
- [ ] Python virtual environment initialized (`python -m venv .venv`)
- [ ] `requirements.txt` created with pinned versions
- [ ] `.gitignore` configured to exclude data, logs, models, and secrets
- [ ] `.env.example` documents all required environment variables
- [ ] Directory structure matches one of the templates above
- [ ] `config.py` or equivalent centralized configuration file created
- [ ] Code formatter configured (`black` for Python)
- [ ] Linter configured (`flake8` or `ruff`)
- [ ] First test written before first feature

## Common architecture mistakes

<AccordionGroup>
  <Accordion title="Starting with microservices">
    Microservices are an organizational solution for teams of 10+ engineers. For a single developer or small team, start with a monolith. Extract services only when you have a proven need for independent deployment or scaling.
  </Accordion>
  <Accordion title="All code in one file">
    A single 2,000-line file is not a project вЂ” it is a draft. Structure forces clarity. If the code cannot be organized into files, the design is not clear enough.
  </Accordion>
  <Accordion title="Architecture chosen for prestige, not maintainability">
    The goal of architecture is to make the system easy to change. Choose the simplest structure that satisfies the actual requirements, not the most sophisticated one you can justify.
  </Accordion>
  <Accordion title="Starting to code before the data flow is clear">
    Identify what data the system receives, what transformations it applies, and what it produces. Draw this on paper before opening an editor. Code written without a clear data model accumulates implicit state that eventually becomes unmanageable.
  </Accordion>
</AccordionGroup>
