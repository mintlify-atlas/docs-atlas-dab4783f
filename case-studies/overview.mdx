Theory is only as good as the projects it produces. The case studies in this section document real software built using the Vibe Coding methodology — with AI as an active pair programmer at every stage, from initial architecture to debugging production bugs. Each case study traces the actual development decisions, the prompts and skills used, the problems encountered, and what was learned.

Reading these case studies gives you something that methodology pages cannot: evidence that the approach works under real constraints, with real data, and with the kind of messy edge cases that do not appear in tutorials.

## What you learn from each case study

Every case study covers the same core questions:

- What was built, and why?
- What technologies were chosen, and how did AI help evaluate trade-offs?
- Where did the development get stuck, and how was it unblocked?
- What prompt patterns and skills proved most useful?
- What would you do differently on the next project?

## Methodology in action

Each project demonstrates a different facet of the Vibe Coding approach:

| Project | Primary technique illustrated |
|---------|-------------------------------|
| Telegram bot | Debugging with precise problem framing; Markdown parsing edge cases |
| Polymarket | Glue coding; ASCII visualization; data pipeline composition |
| Fate Engine | Strong dependency reuse; LLM-driven JSON output; external library integration |

<Note>
  These case studies document work done by real developers using AI as a pair programmer, not AI working autonomously. The human remains the decision-maker; AI accelerates the implementation and helps identify blind spots.
</Note>

## How to read a case study

You do not need to read them in order. If you are building a bot, start with the Telegram case study. If you are doing data analysis, go to Polymarket. If you are integrating external libraries, the Fate Engine study is the most relevant.

Each case study is structured to be useful even if you are not building the same type of project — the prompt patterns, debugging approaches, and architectural decisions transfer broadly.
