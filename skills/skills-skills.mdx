Skills-Skills is the meta-skill of the Vibe Coding CN library — a skill that generates other skills. Give it any domain material (documentation, API specs, a GitHub repository, or a PDF), and it produces a structured, reusable `SKILL.md` that an AI agent can reliably activate and execute. It also refactors existing skills that have become too long, unclear, or prone to misfiring.

## When to use it

Trigger this skill when any of the following applies:

- You want to create a new skill from scratch using docs, specs, or a codebase
- An existing `SKILL.md` is too long, inconsistent, or activates unreliably
- You need to design reliable activation via frontmatter, trigger keywords, and boundary rules
- You want to extract a clean Quick Reference from large documentation
- You need to split long content into navigable `references/` files
- You want a quality gate and validator before shipping a skill

<Note>
  This skill is not a domain skill itself — it builds domain skills. It does not invent external facts. If the source material does not prove a claim, the skill will say so and add a verification path.
</Note>

## Recommended directory layout

```text
skill-name/
├── SKILL.md              # Required: entrypoint with YAML frontmatter
├── references/           # Optional: long-form docs and evidence
│   └── index.md          # Recommended: navigation index
├── scripts/              # Optional: helpers and automation
└── assets/               # Optional: templates and static assets
```

The minimum viable version is a single `SKILL.md`. You can add `references/` later as the skill grows.

## SKILL.md structure

Every generated skill follows this skeleton:

```markdown
---
name: my-skill
description: "[Domain] capability: includes [capability 1], [capability 2]. Use when [decidable triggers]."
---

# my-skill Skill

One sentence that states the boundary and the deliverable.

## When to Use This Skill

Trigger when any of these applies:
- [Trigger 1: concrete task/keyword]
- [Trigger 2]
- [Trigger 3]

## Not For / Boundaries

- What this skill will not do (prevents misfires and over-promising)
- Required inputs; ask 1-3 questions if missing

## Quick Reference

### Common Patterns

**Pattern 1:** one-line explanation
\`\`\`text
[command/snippet you can paste and run]
\`\`\`

## Examples

### Example 1
- Input:
- Steps:
- Expected output / acceptance:

## References

- `references/index.md`: navigation
- `references/...`: long-form docs split by topic

## Maintenance

- Sources: docs/repos/specs (do not invent)
- Last updated: YYYY-MM-DD
- Known limits: what is explicitly out of scope
```

### Frontmatter rules

The YAML frontmatter must satisfy two constraints:

- `name` must match `^[a-z][a-z0-9-]*$` and match the directory name exactly
- `description` must state "what + when" with concrete trigger keywords — not vague help text like "helps with X"

## Generation workflow

The meta-skill enforces a six-step process. Do not skip steps:

<Steps>
  <Step title="Generate a first draft (if source is a docs site, repo, or PDF)">
    Use the bundled Skill Seekers tool to generate a first draft automatically before any manual refinement.

    ```bash
    # Bootstrap dependencies (once)
    ./assets/skills/skills-skills/scripts/skill-seekers-bootstrap.sh

    # Scrape a docs site
    ./assets/skills/skills-skills/scripts/skill-seekers.sh -- scrape \
      --config ./assets/skills/skills-skills/scripts/Skill_Seekers-development/configs/react.json

    # Import into canonical skills tree
    ./assets/skills/skills-skills/scripts/skill-seekers-import.sh react
    ```
  </Step>
  <Step title="Scope">
    Write three sentences: what the skill MUST do, what it SHOULD do, and what it NEVER does.
  </Step>
  <Step title="Extract patterns">
    Pick 10–20 high-frequency patterns — commands, snippets, or flows — that a practitioner would use daily. Each must be directly usable without modification.
  </Step>
  <Step title="Add examples">
    Write at least three end-to-end examples with explicit input, step-by-step actions, and an acceptance criterion.
  </Step>
  <Step title="Define boundaries">
    State what is out of scope and what inputs are required. If inputs are missing, the skill asks 1–3 questions before proceeding.
  </Step>
  <Step title="Split references and apply the gate">
    Move long explanatory text into `references/` and write `references/index.md`. Then run the quality gate checklist and the validator.
  </Step>
</Steps>

## Scaffolding tools

Generate a new skill skeleton from the command line:

```bash
# Full scaffold (production-grade)
./assets/skills/skills-skills/scripts/create-skill.sh my-skill --full --output assets/skills

# Minimal scaffold (quick bootstrap)
./assets/skills/skills-skills/scripts/create-skill.sh my-skill --minimal --output assets/skills
```

Validate an existing skill:

```bash
# Basic validation
./assets/skills/skills-skills/scripts/validate-skill.sh assets/skills/<skill-name>

# Strict validation (enforces all spec requirements)
./assets/skills/skills-skills/scripts/validate-skill.sh assets/skills/<skill-name> --strict
```

## Quality gate

Before shipping any skill, verify all nine points:

<AccordionGroup>
  <Accordion title="Frontmatter and naming">
    `name` matches `^[a-z][a-z0-9-]*$` and matches the directory name. `description` states "what + when" with concrete trigger keywords.
  </Accordion>
  <Accordion title="Activation sections">
    Has "When to Use This Skill" with decidable triggers. Has "Not For / Boundaries" to reduce misfires.
  </Accordion>
  <Accordion title="Quick Reference quality">
    20 patterns or fewer. Each pattern is directly usable (copy/paste ready). Nothing requires paragraph-length explanation — that belongs in `references/`.
  </Accordion>
  <Accordion title="Examples">
    At least three reproducible examples. Each has explicit input, steps, and an acceptance criterion.
  </Accordion>
  <Accordion title="References structure">
    Long content is in `references/`. A `references/index.md` provides navigable links to all files.
  </Accordion>
  <Accordion title="No bluffing">
    Every uncertain claim includes a verification path. Nothing is invented from outside the source material.
  </Accordion>
  <Accordion title="Operator's manual tone">
    Reads like an operator's manual — not a documentation dump. Concise, actionable, and structured.
  </Accordion>
</AccordionGroup>

## Examples

### Example 1: create a skill from documentation

- **Input:** Official API docs + 2–3 real code samples + known failure modes
- **Steps:**
  1. Run `create-skill.sh` to scaffold `assets/skills/<skill-name>/`
  2. Write frontmatter `description` as "what + when"
  3. Extract 10–20 high-frequency patterns into Quick Reference
  4. Add 3+ end-to-end examples with acceptance criteria
  5. Move long content into `references/` and wire `references/index.md`
  6. Run `validate-skill.sh --strict` and iterate
- **Acceptance:** Validator passes in strict mode; skill activates reliably on correct trigger keywords

### Example 2: refactor a "doc dump" skill

- **Input:** An existing `SKILL.md` with large blocks of pasted documentation
- **Steps:**
  1. Identify which parts are short patterns vs. long-form explanation
  2. Move long-form text into `references/` split by topic
  3. Rewrite Quick Reference as copy/paste patterns (20 or fewer)
  4. Add or fix examples until they are fully reproducible
  5. Add "Not For / Boundaries" to reduce misfires
- **Acceptance:** Quick Reference fits on one screen; examples have explicit acceptance criteria

### Example 3: validate and gate an existing skill

- **Input:** `assets/skills/<skill-name>/`
- **Steps:**
  1. Run `validate-skill.sh` (non-strict) to collect warnings
  2. Fix frontmatter/name mismatches and missing sections
  3. Run `validate-skill.sh --strict` to enforce the full spec
  4. Run the scoring rubric in `references/quality-checklist.md`
- **Acceptance:** Strict validation passes with zero errors
